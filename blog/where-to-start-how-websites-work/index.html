New To Programming - How Does A Website Work?

01.08.2021 10:00

One of the hardest questions to start answering as a new developer is “what type of career do you want to pursue?” Many people I talk to (myself included) got into programming because it’s enjoyable, and stimulates our creativity. It’s rewarding and exciting to type some code, hit a button, and see something come into being that didn’t exist 5 minutes ago. However, as a newcomer, it’s difficult to map that impulse and enjoyment to a concrete career path.

There are lots of different types of “computer programmers,” who work on different types of projects, and with different kinds of technology. While building apps and websites is certainly not the only career path – far from it! – it’s a great place for new devs to start to understand how responsibilities can be divided, and what different kinds of careers might look like.

This guide will use a fictional app (for which you can probably guess it’s real-life inspiration) to start to understand how websites work, and the kinds of jobs that go into building them. I’ll also highlight what skills you would want to invest in to pursue each field. Keep in mind:

  1. This guide necessarily cannot cover every type of career, industry, or technology choice. Look at it as a starting point for asking questions about things that interest you.
  2. This guide is not meant to be a full, exhaustive explanation of websites and how they work; I opted for highlighting information that helps us understand the people and groups behind building the app, instead of every single piece of infrastructure.
  3. There is a lot of overlap - all engineers do security to some degree, though Security engineers focus on it. Full stack engineers span the entire “stack” but often each engineer has a favored side of that stack. Think of each of these as flavors you can mix and match, not hard-and-fast career tracks that cannot be combined. Generally, the smaller the company, the more roles you can experiece in one job.
  4. This career field is constantly changing, and your career can constnatly change! If you’re interested in more than one (or all) of these pieces, it’s entirely possible you can work on all of those in your career.

Who Is This Guide For?

This guide is meant for developers just starting to learn how to code, or who are trying to figure out the various career paths in tech and how they relate to each other.

Introducing: Chirpy

Our application, Chirpy, is an app that allows users to create an online profile, post small updates (“chirps”) to their profile, and subscribe to other Chirpy users to read their Chirps.

Yeah… it’s Twitter. Bear with me.

Wait, what do you mean by “app?” Phone apps?

Yes, and no.

Most of us interact with “apps” on our phone, and they’re a thing that we can download from a central repository, like an App Store. You interact with the app by opening it on your phone, and it has nothing to do with other apps on your phone (for the most part).

However, there are also apps that you access every day from within your browser (Chrome, Safari, etc). There is no explicit download step, so it feels very different from the phone experience. Instead of downloading a program to your laptop, you are using your browser to go to a URL that is hosting the app, and you interact with it through your browser, instead of app-specific software. We’ll come back to this later.

The big distinction in terms of “web applications” vs “static websites” is whether the website can be run completely in your browser, or whether it needs to access a database or other backend service to run. For example - if you are able to create an account, or upload data to a website, it’s a web application that has a database containing users' login information somewhere.

In contrast, the website you’re viewing right now is a static website - all of the code that is used to run my blog is downloaded directly to your browser and run there. There are no database items, no interactions with other services. It’s just HTML, CSS, and JS.

Ok, so how does Chirpy work?

Let’s start with your laptop.

Your friend tells you about Chirpy, and convinces you to sign up. So you type “chirpy” into Google, and click on the top result, which is a link to https://chirpy.com. (This is a real website at time of this writing, so we’ll just have to use our imagination).

How does my browser show me this website?

Your browser reads the URL (chirpy.com) and consults a lookup service called DNS, which maps human-readable URLs to more computer-friendly IP addresses. Those look like this: 142.250.69.238.

Each IP address maps to an actual server somewhere, that is hosting the code for the website you’re trying to view. Your browser will follow this IP address and eventually make a request to that server at a particular port number - in this case, port 443. The code running at that location should respond to your browser with a bundle of HTML, CSS, and JS files. These are copied to your browser, and your browser is then responsible for putting those files together and creating the website you can see in your browser.

Jobs

Wow! Already we’ve touched on a ton of different careers, just accessing one website.

Network Engineers

Network Engineers are responsible for all of the pieces of infrastructure that make up the internet. The internet is basically just a giant computer party, where all of the computers know how to talk to one another and how to direct traffic to the right place.

The software that builds the backbone of the internet is built by OS engineers. They make sure that the code that runs the DNS service, routers, and various other pieces of the internet are fast, secure, and reliable.

Open Source Software Developers

If you use Chrome, the code that backs the browser experience was built by open source software engineers. Really, any kind of software engineer can be an “open source engineer” - the difference here is that instead of code being built in a company, and the source code (the human-readable code, as opposed to the 0’s and 1’s the computer cares about) hidden from the public, the source code is “open.” Anyone - you, me, anyone else who is interested - can participate in building this software, or read how it works.

This is very cool, and also is a great learning opportunity. I’ll highlight the ASOS book here - these are free and available online, and walk through various open source projects that you can learn from. I highly recommend them once you get a better feel for the various areas of the tech industry.

Backend Engineers

We will talk a lot more about Backend Engineers, but this is our first encounter with code written by this group of people. The code running at 143.251.xx.xxx and port 443, and is ready to receive and respond to requests, will be written by Backend Engineers. These engineers are responsible for making sure that when you type that URL and hit enter, once your request clears DNS and the Network itself, that you get some sort of response back.

In this case, we made a request to the server to give us HTML, CSS, and JS files that make up a website. For this part of the process, Backend engineers make sure that the server returns the correct files, that it can handle sometimes millions of requests per second efficiently.

Great - now you’re browsing Chirpy. You’re clicking around on the homepage, looking at some example Chirps from famous people, and there is a form you can fill out to sign up for an account.

Frontend Engineer

If a website is a restaurant, the frontend is like the “front of house” – all of the parts of the restaurant that the customer can see and interact with. Frontend Engineers will write the HTML, CSS, and JS that make up the front page experience. They will write code that displays each Chirp correctly, and that creates the signup form.

Frontend Engineers also have a hand in making sure that the various non-code assets, like images and fonts, are served efficiently and displayed correctly. For example, if you’re viewing this website on mobile, you’d probably prefer the website show you a smaller image, because it saves data - no need to download a 5MB file when you can download a 1MB file that looks exactly the same on a small screen. Frontend Engineers make sure that the experience is great for every kind of user.

Web Designer

This role can overlap a lot with Frontend Engineer, but these people are responsible for how things look. They may or may not write code that implements their vision - sometimes, they will design an experience and hand off images, colors, measurements, etc to the frontend engineers to transform into CSS and code that implements that vision.

Designers can decide what color scheme makes the most sense for the website, how large a button should be, what shape it is, etc. They will make sure that design elements are consistent across the entire website expereince, to help users quickly recognize what they’re looking at. A Chirp has a consistent look and feel across the entire experience, whether that’s mobile, desktop, or iPhone app.

Ok, we are convinced we will enjoy using Chirpy, and want to sign up. We start to fill out the form, and then hit Submit.

User Experience / User Interaction Engineer

These Engineers care about how people use a website. Again, Frontend Engineers and Designers care about this too - UX and UI engineers are more technically focused than designers, and more focused on the interaction itself than general frontend engineers.

These engineers might design the signup flow for Chirpy - what fields are on the form on the front page? Does the form submit to the server immediately, or does it walk a user through an entire signup flow before making a request to a server? Appearance is part of this decision, but it’s more focused on a logical decision of how things work.

Backend Engineers

They’re back! (yes, that was a horrible joke). Once the form is submitted, your browser is actually making another request to the server (located at 143.251.xx.xxx, remember?) This time, instead of asking for data, it’s sending data to the server. Specifically, it’s sending your form data (hopefully encrypted) to allow you to sign up for Chirpy.

This form data will be sent to a specific address on the server - an endpoint - that is designed to process that form information. Let’s say you typed in a username you’d like, a password, and some basic user data about yourself. The Backend Engineer would have written the code that looks up the username in the database, and either returns an error saying it’s already been taken, or creates a new User in the database with your information. She or he would also likely have designed the database that holds that information, but in a larger organization, it might be done by a Database Developer or Engineer.

Database Engineer

These engineers care about what format data is stored in, where it’s stored, and for how long (amongst many other things). Think about your User data - you have a username, a password, and some demographic information (age, location, etc).

The database designer is concerned with how easily this information can be found again. When a service like Chirpy has millions of users, it takes time to search through every single one looking for your information - unless we can optimize that process. A database may contain an index, which is a data structure that makes lookups faster. You use an index when you go to a grocery store - you’re looking for a specific item, but the index will show you what kinds of items are in each aisle. The index is the placard hanging from the ceiling, or on the handle of your hsopping cart. While it doesn’t contain information about every single item in the store, it gives you enough to narrow down your search. Database indexes can do even better, and make lookups fast - but they have to be designed thoughtfully.
